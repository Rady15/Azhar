يا هندسة، دي قايمة كاملة بكل الحاجات والـ APIs اللي محتاجينها من جانب الـ Backend لجميع أقسام لوحة تحكم مجمع "أزهر السكني" علشان تكون اللوحة ديناميكية 100% وشغالة ببيانات حقيقية. ياريت نجهز ونضيف الـ Endpoints دي:

==================================================
1. قسم لوحة البيانات والإحصائيات (Dashboard Stats)
==================================================
محتاجين Endpoint رئيسية ترجع أرقام الإحصائيات اللي بتظهر في وش المستخدم أول ما يفتح:
- إجمالي عدد الفلل.
- عدد الفلل المحجوزة والشاغرة.
- إجمالي عدد المستأجرين النشطين.
- عدد طلبات الصيانة المعلقة (Pending).
- الحجوزات النشطة اليوم للمرافق.
- آخر التحديثات والنشاطات الأخيرة في المجمع (عشان تظهر في الـ Recent Activity).
* الـ Endpoint المقترحة: GET /api/Dashboard/stats

==================================================
2. قسم إدارة الفلل (Villas Management)
==================================================
محتاجين CRUD كامل (إضافة وتعديل وحذف وجلب) للفلل:
- جلب كل الفلل (رقم الفيلا، النوع، عدد الغرف، حالة الفيلا: شاغرة/مسكونة، سعر الإيجار السنوي).
- إضافة فيلا جديدة.
- تعديل بيانات فيلا موجودة أو تغيير حالتها.
- حذف فيلا من النظام.
* الـ Endpoints المقترحة:
  * GET /api/Villas
  * POST /api/Villas
  * PUT /api/Villas/{id}
  * DELETE /api/Villas/{id}

==================================================
3. قسم إدارة الشكاوى (Complaints)
==================================================
الشكاوى اللي بيبعتها المستأجرين محتاجين نتحكم فيها كمسؤولين:
- جلب قائمة كل الشكاوى (اسم المستأجر، رقم الفيلا، عنوان الشكوى، التفاصيل، الحالة: قيد الانتظار/تم الحل).
- تحديث حالة الشكوى (مثلاً تحويلها لـ "تم الحل" مع إضافة رد أو ملاحظات من الإدارة).
* الـ Endpoints المقترحة:
  * GET /api/Complaints
  * PUT /api/Complaints/{id}/status

==================================================
4. قسم المدفوعات والتحصيل (Payments & Invoices)
==================================================
لمتابعة الإيجارات والتحصيل المالي:
- جلب قائمة كل الفواتير والمدفوعات (اسم المستأجر، رقم الفيلا، المبلغ، تاريخ الاستحقاق، الحالة: مدفوع/متأخر/قيد الانتظار).
- تسجيل دفعة جديدة (لما المستأجر يدفع كاش أو يحول والمسؤول يأكد الدفع).
- إرسال تذكير بالدفع للمستأجر.
* الـ Endpoints المقترحة:
  * GET /api/Payments
  * POST /api/Payments
  * PUT /api/Payments/{id}/status

==================================================
5. قسم الإعلانات والتعميمات (Announcements / Ads)
==================================================
علشان المسؤول يقدر يبعت تنبيهات أو إعلانات لكل سكان المجمع السكني (زي أعمال صيانة عامة، قواعد جديدة، تعميمات):
- جلب كل الإعلانات النشطة.
- إضافة إعلان جديد (العنوان، المحتوى، تاريخ النشر).
- حذف إعلان قديم أو إنهاء نشاطه.
* الـ Endpoints المقترحة:
  * GET /api/Ads
  * POST /api/Ads
  * DELETE /api/Ads/{id}

==================================================
6. قسم التقارير (Reports & Analytics)
==================================================
لتوليد تقارير دورية للإدارة:
- جلب تقرير مالي (أرباح الإيجارات، المدفوعات المتأخرة خلال فترة معينة).
- جلب تقرير صيانة (عدد الطلبات المكتملة، أكتر فلل طلبت صيانة، التكاليف الإجمالية).
* الـ Endpoints المقترحة:
  * GET /api/Reports/financial
  * GET /api/Reports/maintenance

==================================================
💡 ملاحظات فنية هامة جداً لنجاح الربط:
==================================================
1. تعديل الـ GET Requests اللي بتعتمد على Body:
   ياريت نخلي فلترة طلبات الـ GET (زي تصفية المستأجرين بالإيميل أو الحجوزات) معتمدة على Query Parameters في الـ URL بدلاً من الـ Body، لأن المتصفحات والـ Proxies بتمسح الـ Body في طلبات الـ GET تلقائياً.
   - مثلاً: GET /api/Tenants?email=test@example.com

2. تفعيل الـ CORS بالكامل:
   تأكيد السماح لعنوان الـ Frontend المحلي (زي http://localhost:5173) والـ Domain النهائي بالوصول للسيرفر بدون حظر.

3. توحيد وحفظ الـ Status Enums:
   السماح باستقبال حالات السجلات (الحجوزات، الصيانة، المستأجرين) بصيغ مرنة وتوحيدها.

تسلم إيدك يا هندسة، اللوحة دلوقتي جاهزة تماماً من ناحية الـ Frontend ومستنية الإشارات والبيانات الحقيقية دي علشان تنور وتشتغل بكامل طاقتها!
